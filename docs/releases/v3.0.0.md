# v3.0.0 发布说明

`v3.0.0` 是前后端分离和车辆管控能力完善后的正式版本。与 `v2.0.0` 相比，
本版本不只是设备映射更新，还调整了数据链路、部署方式和停车业务流程。

[查看 v2.0.0...v3.0.0 完整代码差异](https://github.com/yyh111222333/NgDataTwinFrontedDemo/compare/v2.0.0...v3.0.0)

## 主要变化

- 前端通过 Nginx 同源代理统一访问 FastAPI，内网和 WireGuard 使用相同页面效果。
- 新增独立车辆管控页面，接入7至11号门共10个进出通道，12号门不接入。
- 新增车辆进出记录、在场数量、道闸状态、设备在线状态和图片查看。
- 新增车辆快捷预约和车辆档案管理，可下发停车软件白名单。
- 车辆白名单支持蓝牌、黄牌、绿牌，显示同步状态并支持失败重试。
- 有效期内的预约车辆被识别后，由停车软件按照现场车道规则自动放行。
- 7、8、10号门补充远程开闸控制能力；9、11号门保留适配后的设备接口。
- 智慧监控概况、行车监测概况和车辆在场数据接入子系统实时数据。
- 右上角接入南京实时天气，服务器状态改为 FastAPI 健康检查。
- 实时更新使用 SSE，断线时自动重连并降级为定时查询。
- 前端移除 MQTT 运行依赖，业务数据统一使用 REST 和 SSE。

## 部署变化

- 前端镜像升级为 `ngdtdemo-frontend:3.0.0`。
- FastAPI 默认上游为 `http://192.168.10.11:19080`，可通过 `.env` 覆盖。
- Nginx 新增 `/api/`、`/parking-api/` 和 `/health` 同源代理。
- `docker-compose.yml` 只包含前端服务，不包含 FastAPI、Redis、PostgreSQL或MQTT。
- 本版本不会管理或修改宿主机 `5000` 端口上的现有容器。

## 升级注意事项

1. 先部署配套 FastAPI 后端并完成数据库迁移。
2. 根据 `.env.example` 配置全部上游地址。
3. 保持 `VITE_API_BASE_URL` 为空，避免将内网或WG地址写死到前端包中。
4. 使用 `docker compose up -d --build --force-recreate frontend` 重建前端。
5. 升级后检查 `/health`、`/cockpit` 和 `/parking`。

完整安装、升级与回滚步骤见仓库根目录
[README](https://github.com/yyh111222333/NgDataTwinFrontedDemo/blob/v3.0.0/README.md)。
